## Overview

Subtasks allow you to break down complex tasks into smaller, manageable pieces. Each subtask belongs to a parent task and can have its own status, description, tags, and completion tracking. Subtask completion automatically contributes to the parent task's progress percentage.

## Subtask Model

The Subtask entity structure:

```kotlin
@Entity
@Table(name = "subtasks")
data class Subtask(
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    val subtaskId: Long? = null,
    
    val taskId: Long,
    val title: String,
    val description: String? = null,
    val durationTagId: Int? = null,
    val status: Status = Status.PENDING,
    val completedAt: LocalDateTime? = null,
    val displayOrder: Int = 0,
    val createdAt: LocalDateTime? = null,
    val updatedAt: LocalDateTime? = null,
    
    var task: Task? = null,
    var tags: Set<Tag> = emptySet()
)
```

## Subtask Properties

### Core Properties

| Property | Type | Description |
|----------|------|-------------|
| `subtaskId` | Long | Unique identifier (auto-generated) |
| `taskId` | Long | ID of the parent task (required) |
| `title` | String | Subtask title (required, max 255 chars) |
| `description` | String | Detailed description (optional, text) |

### Status & Tracking

| Property | Type | Description |
|----------|------|-------------|
| `status` | Enum | Subtask status: PENDING, IN_PROGRESS, COMPLETED |
| `completedAt` | LocalDateTime | Timestamp when subtask was completed |
| `displayOrder` | Int | Order for sorting subtasks (default: 0) |

### Time Estimation

| Property | Type | Description |
|----------|------|-------------|
| `durationTagId` | Int | Reference to duration estimation |
| `createdAt` | LocalDateTime | Subtask creation timestamp |
| `updatedAt` | LocalDateTime | Last modification timestamp |

### Relationships

| Property | Type | Description |
|----------|------|-------------|
| `task` | Task | Parent task this subtask belongs to |
| `tags` | Set&lt;Tag&gt; | Associated tags for categorization |

## Creating a Subtask

Create a new subtask under an existing task:

```bash
curl -X POST https://api.taskpro.com/api/subtasks \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
  -d '{
    "taskId": 42,
    "title": "Design wireframes",
    "description": "Create low-fidelity wireframes for main user flows",
    "tagIds": [2],
    "displayOrder": 1
  }'
```

### Request DTO

```kotlin
data class SubtaskCreateDto(
    val taskId: Long,
    val title: String,
    val description: String? = null,
    val tagIds: List<Long>? = null,
    val durationTagId: Int? = null,
    val status: String = "pending",
    val displayOrder: Int = 0
)
```

## Updating a Subtask

Update subtask properties including status:

```bash
curl -X PUT https://api.taskpro.com/api/subtasks/15 \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
  -d '{
    "status": "completed"
  }'
```

### Update DTO

```kotlin
data class SubtaskUpdateDto(
    val title: String? = null,
    val description: String? = null,
    val tagIds: List<Long>? = null,
    val durationTagId: Int? = null,
    val status: String? = null,
    val displayOrder: Int? = null
)
```

<Note>
  When you set status to "completed", the `completedAt` timestamp is automatically set to the current time.
</Note>

## Deleting a Subtask

Permanently delete a subtask:

```bash
curl -X DELETE https://api.taskpro.com/api/subtasks/15 \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"
```

<Warning>
  Deleting a subtask affects the parent task's completion percentage. This action cannot be undone.
</Warning>

## Task Progress Tracking

Subtasks automatically contribute to the parent task's progress. When you retrieve a task, it includes progress information:

```bash
curl -X GET https://api.taskpro.com/api/tasks \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"
```

### Response with Progress

```json
[
  {
    "taskId": 42,
    "title": "Design new landing page",
    "status": "in_progress",
    "totalSubtasks": 3,
    "completedSubtasks": 1,
    "completionPercentage": 33.33,
    ...
  }
]
```

### Progress Calculation

```kotlin
val completionPercentage = if (totalSubtasks > 0) {
    (completedSubtasks.toDouble() / totalSubtasks) * 100
} else {
    null
}
```

## Subtask Status Workflow

Typical workflow for subtask status changes:

<Steps>
  <Step title="Created (PENDING)">
    New subtask starts with status "pending"
  </Step>
  <Step title="Started (IN_PROGRESS)">
    Update status to "in_progress" when work begins
  </Step>
  <Step title="Completed (COMPLETED)">
    Mark as "completed" when finished - `completedAt` is automatically set
  </Step>
</Steps>

## Subtask Relationships

### Many-to-One: Subtask → Task

Each subtask belongs to exactly one task:

```kotlin
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "task_id", insertable = false, updatable = false)
var task: Task? = null
```

The parent task has cascade deletion enabled:

```kotlin
// In Task.kt
@OneToMany(mappedBy = "task", cascade = [CascadeType.ALL], orphanRemoval = true)
var subtasks: Set<Subtask> = emptySet()
```

<Warning>
  When a task is deleted, all its subtasks are automatically deleted due to `CascadeType.ALL` and `orphanRemoval = true`.
</Warning>

### Many-to-Many: Subtask ↔ Tag

Subtasks can have multiple tags:

```kotlin
@ManyToMany
@JoinTable(
    name = "subtask_tags",
    joinColumns = [JoinColumn(name = "subtask_id")],
    inverseJoinColumns = [JoinColumn(name = "tag_id")]
)
var tags: Set<Tag> = emptySet()
```

## Best Practices

<Steps>
  <Step title="Keep subtasks focused">
    Each subtask should represent a single, actionable piece of work.
  </Step>
  <Step title="Use appropriate granularity">
    Aim for 3-7 subtasks per task. Too few and you're not breaking down enough; too many and you're over-engineering.
  </Step>
  <Step title="Order logically">
    Use `displayOrder` to arrange subtasks in the sequence they should be completed.
  </Step>
  <Step title="Track time estimates">
    Use `durationTagId` to estimate how long each subtask will take.
  </Step>
  <Step title="Tag by type">
    Use tags to categorize subtasks by domain (frontend, backend, design, testing).
  </Step>
  <Step title="Update status regularly">
    Keep subtask status current to accurately reflect progress on the parent task.
  </Step>
</Steps>

## Display Order

Subtasks support custom ordering through the `displayOrder` property:

```bash
# Reorder subtasks
curl -X PUT https://api.taskpro.com/api/subtasks/15 \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
  -d '{
    "displayOrder": 5
  }'
```

<Tip>
  When displaying subtasks, sort by `displayOrder` ascending, then by `subtaskId` for consistent ordering.
</Tip>

## Limitations

<CardGroup cols={2}>
  <Card title="No Nesting" icon="ban">
    Subtasks cannot have their own subtasks. Keep the hierarchy at two levels: Task → Subtask.
  </Card>
  <Card title="Single Task Parent" icon="link">
    A subtask can only belong to one task. It cannot be shared across multiple tasks.
  </Card>
  <Card title="No Direct User Association" icon="user">
    Subtasks are associated with users through their parent task, not directly.
  </Card>
  <Card title="Cascade Deletion" icon="trash">
    Deleting a task deletes all its subtasks. Consider archiving tasks instead.
  </Card>
</CardGroup>
